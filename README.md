> BDD Gherkin scenarios, custom fixtures for page object injection, storage-state session reuse,  
> JSON-driven data management, Allure reporting, and Docker-based CI/CD on GitHub Actions and Azure DevOps.

## 🏗️ Architecture

### Layer Responsibilities

| Layer                | Files                                               | Responsibility                                                                      |
|----------------------|-----------------------------------------------------|-------------------------------------------------------------------------------------|
| **Fixtures**         | `src/fixtures/fixtures.ts`                          | Extends `base.extend()` — injects page objects into every step via destructuring    |
| **Page Objects**     | `src/pages/*.ts`                                    | Locators + reusable async methods. No test logic — pure UI interaction              |
| **Step Definitions** | `tests/steps/*.ts`                                  | Maps Gherkin to page object calls. Manages `currentEmployee` state and `users.json` |
| **Feature Files**    | `tests/features/*.feature`                          | Business-readable Gherkin. Tagged with `@smoke`, `@regression`, `@createEmp` etc.   |
| **Utils**            | `src/utils/*.ts`                                    | Session management, Faker data, JSON persistence, Excel, file operations            |
| **Config**           | `config/env.config.ts`                              | URLs, credentials, session paths, timeouts — single source of truth                 |
| **Global Setup**     | `config/global.setup.ts`                            | Runs once before all tests — logs in per user type, saves `.sessions/*.json`        |
| **CI/CD**            | `playwright.yml` + `azure-pipelines.yml`            | GitHub Actions (Docker) + Azure DevOps (Docker + Key Vault)                         |

## 🔐 Session Management & Storage State

Tests avoid repetitive UI login by reusing saved browser sessions via Playwright's `storageState`.

### How It Works

```
global.setup.ts runs ONCE before all tests
    ↓
session_util.saveSession(browser, 'superadmin')
    → navigates to login page
    → fills credentials from env.config.ts → CREDENTIALS['superadmin']
    → clicks Login, waits for dashboard
    → context.storageState({ path: '.sessions/superadmin.json' })  ← saves cookies + localStorage
    ↓
playwright.config.ts — each project loads the saved session:
    smoke project:       use: { storageState: SESSION_PATHS['superadmin'] }
    sanity project:      use: { storageState: SESSION_PATHS['superadmin'] }
    regression projects: use: { storageState: SESSION_PATHS['superadmin'] }
    ↓
Each test starts already authenticated — login.steps.ts just navigates to /dashboard
```

### Session Lifecycle

| Environment | Created | Reused | Deleted |
|---|---|---|---|
| **Local** | Once per run (skipped if file exists) | All subsequent tests | Stays on disk |
| **CI** | Every pipeline run (fresh runner) | All tests in that run | When runner terminates |
| **Git** | Never committed | — | `.sessions/` is git-ignored |

### Credential Flow

```
Local:        .env file       → dotenv → process.env → CREDENTIALS['superadmin']
GitHub CI:    GitHub Secrets  → env vars → process.env → CREDENTIALS['superadmin']
Azure DevOps: Key Vault       → Variable Group → env vars → CREDENTIALS['superadmin']
```

### Multi-User Support
- `superadmin` — active now: session created, loaded by all current projects
- `clientadmin` — ready: session creation guarded by `if (CREDENTIALS['clientadmin'].username)` — skipped until `CLIENTADMIN_USER` secret is set
- `ess` — same pattern as clientadmin — add secret when ESS tests are created

## 👤 User Creation and Role Management Flow

This is the most architecturally significant part of the project — it mirrors how a real HR admin sets up users in OrangeHRM.

### Employee Types

| Type          | JSON Key                | Description                                           |
|---------------|-------------------------|--------------------------------------------------------|
| `superadmin`  | `users.superadmin`      | Hard-coded demo admin — `Admin / admin123`             |
| `clientadmin` | `users.clientadmins[0]` | PIM employee promoted to Admin role                    |
| `ess`         | `users.essUsers[0]`     | PIM employee assigned ESS (Employee Self Service) role |

**************************************************************************************************************************************
### Step-by-Step Flow

#### Step 1 — Create Employees in PIM (`@createEmp`)
- Scenario Outline runs **twice** — once for `clientadmin`, once for `ess`
- `FakerDataUtil.getEmployeeId()` generates a unique `EMP-XXXXXX` ID at runtime
- Employee details filled into the Add Employee form
- On save → `saveEmployeeByType()` writes to `users.json` at `[0]` position (replace, not push — prevents duplicate accumulation)

#### Step 2 — Validate in Employee List (same scenario)
- Reads saved `employeeId` from `users.json` as ground truth
- Pagination-aware loop: reads `nth(1)=ID`, `nth(2)=First(&Middle)Name` per row
- `waitForTableToLoad()` called **after** `paginationNext.click()` — trace confirmed calling it before caused page 2 to be skipped

#### Step 3 — Assign Roles and Credentials in Admin (`@employeeRoleMapping`)
- DataTable: `role | status | employee Name | username | password`
- For each row: Add button → User Role dropdown → Status → employee autocomplete → fill credentials
- `saveAdminCredentials()` finds the employee in `users.json` by `firstName+lastName` and appends `adminCredentials: { username, password }`

#### Step 4 — Login as clientadmin or ess
- `Given user login as "clientadmin"` reads `users.json` → `clientadmins[0].adminCredentials`
- Zero hardcoded credentials in step code

**************************************************************************************************************************************
### users.json — State After Each Stage
----json
// Default (empty)
{ "clientadmins": [], "essUsers": [] }

// After @createEmp
{ "clientadmins": [{ "firstName": "Dipu", "middleName": "Ak",

// After @employeeRoleMapping
{ "clientadmins": [{ ...employee, 
                      "adminCredentials": { "username": "dipu17admin", "password": "Snow@110029" } }] }

## 🔧 Custom Fixtures (Dependency Injection)

The project uses `base.extend<MyFixtures>()` — Playwright's recommended DI pattern for playwright-bdd:
```ts
// fixtures.ts
export const test = base.extend<MyFixtures>({
  loginPage:     async ({ page }, use) => { await use(new LoginPage(page)); },
  dashboardPage: async ({ page }, use) => { await use(new DashboardPage(page)); },
  adminPage:     async ({ page }, use) => { await use(new AdminPage(page)); },
  pimPage:       async ({ page }, use) => { await use(new PIMPage(page)); },
});

// Steps receive page objects via destructuring — no manual instantiation
When('user navigates to "PIM" module', async ({ dashboardPage, pimPage }) => {
  await dashboardPage.pim.click();
  await expect(pimPage.pimHeader).toBeVisible();
});
```

Each fixture is created fresh per scenario — complete isolation in parallel runs.

## 🐛 Notable Framework Bugs Fixed

| Bug | Root Cause | Fix |
|--------------------------------------------------------------|-------------------------------|---------------------------------------------------------------|
| `waitForTableToLoad()` called before pagination click | Timing — page 2 not yet loaded | Moved to **after** click — trace confirmed page 2 count was 0 |
| Loop variable `i` not used in row iteration | Off-by-one indexing | Changed to `.nth(i)` to scope cell reads to correct row |
| `${savedEmp}` printed `[object Object]` | Object serialisation | Fixed to `${savedEmp.employeeId}` |
| Column `nth(2)` matched against `firstName+lastName` | Wrong column assumption | Fixed to `firstName+middleName` (actual column content) |
| `addButton.click()` outside DataTable loop | Loop scope error | Moved inside loop — ESS row never had a form opened |
| Circular import: `MyInfoPage.ts` imported from `fixtures.ts` | Dependency cycle | Removed all step/fixture imports from page objects |
| ESS users accumulating via `.push()` on every run | Unbounded array growth | Changed to `[0]=` replacement for both types |
| `bddTestData not found` on CI — line number mismatch | `playwright 1.61.0-alpha` pulled in by `@playwright/cli` conflicted with `@playwright/test 1.60.0` on Linux module resolution | Removed `@playwright/cli`, pinned `playwright: 1.60.0` in devDependencies |
| CRLF line endings in `.feature` files causing bddgen drift on Linux | No `.gitattributes` — Windows CRLF committed | Added `.gitattributes` enforcing LF; `playwright-bdd.config.ts` `importTestFrom` locked entry point |
| `.features-gen/` stale committed files conflicting with fresh CI generation | `.features-gen/` not in `.gitignore` | Added to `.gitignore`; untracked with `git rm --cached -r .features-gen/` |
